# Thermo-Nuclear Code Quality Review — Round 2

## ✅ RESOLVED: Finding 1 (file size)

`main.go` is now 5 lines. Implementation is in 7 focused files across `internal/app/`. Largest file is `keychain.go` at ~370 lines. Clean.

## ✅ RESOLVED: Finding 2 (HOTP counter offset bug)

`insertKey()` now calls `c.parse(c.data)` instead of `c.parse([]byte(line))`.
Test `TestInsertKey_HOTPOffsetInNonEmptyKeychain` guards against regression.

## ✅ RESOLVED: Finding 3 (code() had hidden I/O side effects)

`code()` is now a pure getter. HOTP counter advancement and file persistence moved
to `incrementCounter()`. Both `show()` and `showWithClip()` call `code()` then
`incrementCounter()`. Tests updated to reflect the split.

## ✅ RESOLVED: Finding 4 (duplicated drift-tolerance logic)

Extracted `totpWithFallback(key []byte, digits int) int` into `totp.go`.
Both `code()` and `showAll()` now call it instead of inlining the ±1 window
logic. Three tests cover the helper: returns a code, stable within window,
and matches direct `totp()` call.

## ✅ RESOLVED: Finding 5 (showWithClip re-opened keychain)

`showWithClip()` was inlined into `menu()`, which already has a loaded
`*Keychain` as its receiver. No more unnecessary re-read from disk.
`TestKeychainCode_AfterLoad` guards against regression.

## ✅ RESOLVED: Finding 6 (minor nits)

- **6a** — `noSpace()` kept as-is (named function reads better than inlining)
- **6b** — `WriteString` + `\n` replaced with `fmt.Fprintf` in menu View;
  `TestMenuView_ContainsKeyNames` guards rendering output

  preventing any state inconsistency from being used
